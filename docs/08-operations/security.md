## Secrets Manager layout

All secrets under `/connector/{env}/` in AWS Secrets Manager. Access via IAM role `connector-{env}-role`.

```
/connector/prod/
├── rds/
│   └── credentials                          { username: "connector_user", password: "<pwd>" }
├── platform-apps/
│   ├── meta                                 { client_id, client_secret, app_secret, webhook_verify_token }
│   ├── youtube                              { client_id, client_secret, gcp_project_id }
│   ├── twitch                               { client_id, client_secret }
│   └── tiktok                               { client_key, client_secret }
├── outbound-hmac-secrets                    { active: "<new>", previous: "<old>" }
├── inbound-webhook-secrets/
│   ├── meta                                 { app_secret: "<same as platform-apps>" }
│   ├── youtube                              { pubsubhubbub_hub_secret: "<rotatable>" }
│   └── twitch                               { per_subscription_secrets: {...} }
├── service-tokens/
│   └── backend-api                          { active: "<new>", previous: "<old>" }
└── admin-tokens                             { tokens: [{ token: "<t>", label: "ops-<op>", expires_at: "2026-12-31" }, ...] }
```

All secret values are randomly generated (≥ 32 bytes), never derived from anything human-memorable.

**Rotation cadence:**
- Platform app credentials: annually, or immediately on suspected leak
- Outbound HMAC: quarterly
- Service tokens: semi-annually
- Admin tokens: per-operator, rotated on personnel change
- RDS password: annually

**IAM permissions:** only the connector EC2 instance role can read these. Operators with `SecurityAdmin` role can rotate.

## KMS envelope encryption for OAuth tokens

Purpose: protect refresh/access tokens at rest, decryption is auditable via CloudTrail.

```
     Plaintext access_token
             │
             ▼
  [data key generated by KMS — random AES-256]
             │
             ▼
     AES-GCM encrypt → access_token_ciphertext
             │
             ▼
  [data key itself encrypted by KMS CMK]
             │
             ▼
     Stored in DB:
       • access_token_ciphertext (BLOB)
       • kms_data_key_ciphertext (BLOB)
```

KMS CMK alias per env: `alias/connector-{env}-token`. One CMK per env; not rotated on schedule (rotated only on security events).

**On read:**
1. Fetch row from `oauth_tokens`.
2. Decrypt `kms_data_key_ciphertext` via KMS `Decrypt` API → plaintext data key in memory.
3. AES-GCM decrypt `access_token_ciphertext` with data key → plaintext token in memory.
4. Use token for the API call.
5. Zero out the plaintext data key immediately (overwrite buffer).

Tokens never logged, never returned in API responses, never in error messages.

**Envelope key rotation:**
- KMS CMK rotation enabled (annually by default).
- Re-encrypt all tokens with new CMK via nightly cron `connector-cli tokens rekey --env prod`. Runs over all `oauth_tokens` rows touching one at a time, bounded rate.
- Old CMK versions retained for decryption compatibility per AWS policy.

## HMAC multi-secret rotation (outbound events)

Described in [`../06-event-catalog.md`](../06-event-catalog.md). Procedure in [`runbook.md`](runbook.md) §Rotate HMAC.

Key points:
- Two slots: `active` + `previous`.
- Signer uses `active` for new signatures.
- Verifier (subscriber) accepts both `active` AND `previous`.
- Rotation is additive — promote new to active, demote current to previous, null out old-previous.
- Zero downtime. Cycle once per quarter.

Secret set shape in Secrets Manager:
```
{ "active": "<base64-32-random>", "previous": "<base64-32-random>" | null }
```

**Signature algorithm** (shared detail across outbound and inbound verification):
```
signature = HMAC_SHA256(secret, signature_timestamp + "." + raw_body)
header    = "sha256=" + hex(signature)
```

Timestamp is ISO-8601 UTC, included in the envelope. Signatures older than 10 min are rejected (replay protection).

Constant-time comparison mandatory at verification.

## Service tokens (backend-api → connector auth)

Long-lived tokens, `Authorization: Service-Token <token>`. Multi-secret rotation like HMAC:

```
/connector/prod/service-tokens/backend-api  →  { "active": "<t1>", "previous": "<t2>" }
```

- Connector's auth middleware accepts either.
- backend-api updates its env var to `active`, deploys, confirms it works.
- Ops demotes `active` to `previous`, nulls out old-previous. Rotation complete.

If a service token leaks: rotate immediately; revoke the leaked slot by setting both `active` and `previous` to new values; force backend-api redeploy.

## Admin tokens (human operators → connector admin API)

Per-operator tokens stored in Secrets Manager under `/connector/prod/admin-tokens`. Each carries:
- `token` — the bearer value
- `label` — operator identifier
- `expires_at` — ISO date, typically 6 months out

Connector's admin middleware validates:
1. Token is in the active set.
2. Token has not expired.
3. Operator's email (via audit log) matches their Active Directory / Google Workspace.

Revoked by removing from the set. No long-term retention beyond expiry.

## Audit log

Every privileged action lands in `audit_log` (see [`../04-data-model.md`](../04-data-model.md)). Never pruned. Retention: **permanent**.

Examples captured:
- Any admin API call (`actor` = operator label or `'service:backend-api'`)
- Token rotations (active → previous → null)
- KMS Decrypt calls via CloudTrail, separately
- GDPR purges
- Cadence default changes
- Tier changes per account
- Disconnect/reconnect

**Audit log is the compliance artifact.** Separate from ops logs (Loki, 30-90d). It outlives ops logs on purpose.

## OAuth state management

Generated per connect flow, stored in Redis + MySQL `pending_connections`:
- Format: opaque 32-byte base64url string
- Bound to `(platform, user_id, organization_id)` — modifying any field invalidates
- TTL: 10 minutes
- Single-use: successful callback deletes the state; second callback with same state → 410 Gone
- Never logged; never exposed in error messages

**Attack surface:** state parameter prevents CSRF on the callback. If state is missing/invalid/expired, the callback refuses to process.

## TLS and network

- All inbound to connector: HTTPS via nginx with Let's Encrypt (existing certs).
- All service-to-service: HTTPS or private network.
- RDS: in-VPC, TLS enforced via connection string (`?ssl=true`).
- Redis: in-VPC, no TLS in launch config (acceptable within VPC; enable if shared-tenant ever introduced).
- S3: TLS enforced via bucket policy (`aws:SecureTransport = true`).

No public endpoints other than:
- `/oauth/callback/:platform` (browser redirects, no secrets in body)
- `/webhooks/ingest/:platform` (platform webhooks, signature-verified)
- `/healthz`, `/readyz` (no sensitive data)

Internal API `/v1/*` is VPC-private; ALB has no public DNS.

## GDPR data subject rights

See [`runbook.md`](runbook.md) §GDPR purge for operational procedure.

Connector commitments:
- **Deletion within 24h** of request (NF-62).
- **Export**: `GET /v1/admin/accounts/:id/export` returns a JSON dump of all data the connector holds about that account, signed by operator. Consumed by backend-api which combines with its own export.
- **Consent records** live in `audit_log` entries of type `consent_captured` — include scope set, privacy notice version, timestamp, organization that initiated. Retained permanently for compliance.

## Vulnerability response

- Dependency scanning via Dependabot (existing GitHub setup).
- Container image scanning on every ECR push (AWS Inspector).
- Severity triage:
  - CRITICAL/HIGH: patch within 48h, deploy out-of-band
  - MEDIUM: patch within 2 weeks
  - LOW: next regular cycle
- Security incidents: follow the broader engineering security playbook.
